# 摸金虾 架构决策记录（ADR）

## ADR-001: 策略文件作为唯一数据总线

**日期**: 2026-05-28  
**状态**: ✅ 已采纳  
**版本**: v0.2

### 背景

v0.1 阶段各 CRON 独立运行，没有跨 CRON 状态共享机制。收盘复盘写腾讯文档，盘前分析独立拉数据，两者不互通。

### 决策

采用每日独立策略文件（`strategies/{date}.json`）作为所有 CRON 间的唯一数据交换媒介。`strategy_index.json` 作为索引指针。

### 替代方案

- `context_from` 注入：被否决，旁路传递、非结构化、不可审计
- 共享数据库：过度设计，7 个 CRON 不需要

## ADR-002: risk_state 确定性计算

**日期**: 2026-05-28  
**状态**: ✅ 已采纳  
**版本**: v0.2

### 背景

LLM 自行判断风险等级导致不一致：同一账户状态，不同模型/不同时刻可能给出不同 risk_state。

### 决策

`calc_risk_state.py` 硬编码五级规则（normal→cautious→defensive→emergency），所有 CRON 必须调用此脚本，禁止 LLM 手动覆盖。

### 教训

2026-06-02 事故：CRON prompt 缺少「禁止 LLM 覆盖」规则，导致 `consecutive_loss_days` 跨交易日不归零，熔断误持续 2 天。已在 prompt 中加固。

## ADR-003: 侦察收敛到午后单节点

**日期**: 2026-06-01  
**状态**: ✅ 已采纳  
**版本**: v0.4

### 背景

v0.3 阶段午盘(10:00)和尾盘(14:35)各自做独立市场侦察（大盘/板块/资金流），逻辑复制两份，且午后检查(13:16)只能写文本预警不能更新策略。

### 决策

三层模型：策略层定制蓝图，侦察层（午后 13:16）集中侦察并写结构化 `market_snapshot`，执行层（午盘/尾盘）只读快照 + 一次 API 验证。

### 收益

侦察次数从 4 次/天降至 3 次，信息传递从「文本→LLM 重解释」变为「结构化 JSON→直接匹配」。

## ADR-004: 引擎分离，策略文件连接

**日期**: 2026-06-02  
**状态**: ✅ 已采纳  
**版本**: v1.0

### 背景

v0.4 阶段引擎一结果通过 `context_from` 注入尾盘执行，绕过了策略文件。引擎二从未成功运行。

### 决策

引擎一 → 写入 `candidate_pool.engine1`。引擎二 → 读取 engine1 → 写入 `candidate_pool.engine2.ranked`。尾盘/收盘从策略文件读取候选池。`context_from` 全部清零。

### 理由

每个引擎方法不同、脚本独立、可独立失败。策略文件做总线保证可审计、可回溯。

## ADR-005: 报纸只用于锚点时刻

**日期**: 2026-06-02  
**状态**: ✅ 已采纳  
**版本**: v1.0

### 背景

v0.4 阶段盘前和尾盘都用报纸图片，但尾盘距收盘仅 25 分钟，出图耗时且信息密度不如文本。

### 决策

报纸图片仅用于两个锚点：开盘前（盘前早参）和收盘后（收盘复盘）。盘中全部用精简文本。

## ADR-006: 设计文档与操作手册分离

**日期**: 2026-06-02  
**状态**: ✅ 已采纳  
**版本**: v1.0

### 背景

`moni-sim-trading` skill 混合了两种内容：给 Hermes 看的操作步骤（怎么改 CRON、怎么排查故障）和给人看的产品设计（changelog、roadmap）。

### 决策

前缀分离：`references/design-*.md` 为产品设计文档（changelog、roadmap、ADR），不带前缀的为操作参考。SKILL.md 顶部明确两条阅读路径。
