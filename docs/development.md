# 摸金虾 开发手册

> v1.0.1 · 2026-06-03

本文档定义摸金虾的全生命周期流程：需求 → 迭代 → 开发 → 发布 → 运维。

---

## 一、需求管理

### 1.1 从想法到规格

```
想法 ──→ 需求清单.md（分配#号，状态=待排期）
              │
              ├── 开始讨论设计 ──→ 需求池/SRS-NNN（状态=设计中）
              │                         │
              │                    方案定稿（状态=待实施）
              │                         │
              │              迭代计划选中（状态=实施中）
              │                         │
              │                  实施完成 → SRS 更新日志闭合
              │
              └── 讨论后关闭 ──→ 状态=已关闭，写原因
```

关键点：
- **需求清单#号是唯一 ID**，SRS 编号继承它——`需求清单#003 → SRS-003`，同一编号体系
- **SRS 不等于开始实施**——方案讨论就可以建 SRS（如 SRS-002 当前状态=设计中），选中后才是实施

### 1.2 SRS 生命周期状态

| 状态 | 含义 | 何时进入 |
|------|------|------|
| 设计中 | 方案讨论，未定稿 | 需求清单条目开始讨论时 |
| 待实施 | 方案定稿，等排期 | 方案设计完成、风险已评估 |
| 实施中 | 本轮迭代在做 | 迭代计划选中 |
| 已完成 | 上线，更新日志闭合 | 发布 checklist 通过 |

需求清单条目的状态字段与 SRS 同步更新。

### 1.3 操作规则

| 时机 | 操作 |
|------|------|
| 有新想法 | 需求清单新增一行，分配#号，状态=待排期 |
| 开始讨论设计 | 建 `需求池/SRS-NNN-description.md`（五段模板），状态=设计中 |
| 方案定稿 | SRS 状态→待实施 |
| 迭代选中 | SRS 状态→实施中，迭代计划写条目 |
| 实施完成 | SRS 更新日志记录 + 状态→已完成 |
| 想法被否 | 需求清单状态→已关闭，写原因；SRS 若已存在则同步关闭 |

### 1.4 SRS 文件规范

- 命名：`需求池/SRS-NNN-description.md`，NNN=需求清单#号，三位补零
- 结构：业务需求 → 方案设计 → 风险与缓解 → 关联 → 更新日志

## 二、问题管理

### 2.1 唯一问题源

```
盯盘红灯 / 用户提出 / 审计报警 ──→ 问题清单.md（唯一追踪源）
                                         │
                                   按严重度分流：
                                   P0 → 立即修，关条目
                                   P1 → 周六修，关条目
                                   P2 简单 → 周六修，关条目
                                   P2 需设计 → 关条目，同时建需求清单条目 → SRS
```

所有问题先登记问题清单。没有第二个地方。

### 2.2 严重度定义

| 等级 | 含义 | 响应 | 例子 |
|:--:|------|:--:|------|
| 🔴 P0 | 影响当天交易正确性 | **立即修**，不等到周六 | CRON 挂了、策略文件损坏、硬约束失效 |
| 🟡 P1 | 影响近期交易质量 | 周六集中修 | Scenario 触达率低、risk_state 断层 |
| 🟢 P2 | 体验/远期改进 | 周六修；如涉及方案设计则关条目→建需求清单→SRS | 消息格式优化、审计趋势图 |

### 2.3 状态定义

| 状态 | 含义 |
|:--:|------|
| 待排期 | 已登记，等周六或下次迭代 |
| 已排期 | 已纳入本周六或当前迭代 |
| 已关闭 | 已修复，填修复日期 |

### 2.4 问题清单条目格式

| # | 状态 | 严重 | 问题 | 现象 | 来源 | 关联 |
|:--:|:--:|:--:|------|------|:--:|------|
| 001 | 已排期 | 🟡 P1 | Scenario 触达率 21% | 引擎选了买不进 | 审计 | SRS-001 |

### 2.5 操作规则

| 时机 | 操作 |
|------|------|
| 盯盘发现红灯 | 问题清单加一行，严重按上表判定，来源=盯盘 |
| 审计报警 | 问题清单加一行，来源=审计 |
| 用户提出问题 | 问题清单加一行，来源=用户 |
| P0 问题 | 立即修 → 修完状态→已关闭 |
| 周六排期 | 从问题清单挑 P1/P2：P1→直接修关条目；P2 简单→直接修关条目；P2 需设计→关条目，同时建需求清单+SRS |
| 修完 | 状态→已关闭，填修复日期

## 三、迭代管理

### 3.1 三层节奏

| 层级 | 频率 | 做什么 | 不改什么 |
|------|------|------|---------|
| **盯盘** | 每日 15:05 后 | 三信号检查，红灯记问题清单 | 不改代码 |
| **迭代** | 每周六 | 跑审计 + 排下周 + 集中改 | — |
| **大版本** | 每月末 | 月目标核算 + 回溯 + roadmap 刷新 | — |

### 3.2 每周六流程

1. **计分卡**：`python3 ~/.hermes/scripts/balanced_scorecard --format markdown`
2. **Scenario 审计**：`python3 ~/.hermes/scripts/scenario_audit.py`
3. **排期**：打开 `迭代计划.md`——
   - 从 `需求清单.md` 挑"待实施"的条目 → SRS 状态→实施中
   - 从 `问题清单.md` 挑 P1/P2 → 状态→已排期
   - 更新迭代计划，写本周要做的
4. **集中改**：改 CRON prompt → 改脚本 → 更新 skill → `cronjob run` 验证
5. **关问题**：本周修完的问题条目状态→已关闭
6. **补记 P0**：本周如有 P0 已修，问题条目补关 + 严重变更记 `changelog.md`

### 3.3 迭代计划格式

```
# 迭代 N · 日期范围

## 来自需求清单
- [ ] #001 引擎路径闭环 → SRS-001

## 来自问题清单
- [ ] #001 Scenario 触达率 21% → 直接修
- [ ] #005 risk_state 断层 → 直接修

## 本周不排
- #003 引擎三 — 阻塞于引擎一/二稳定
```

### 3.4 盘中安全规则

- **09:00-15:05 不改任何 CRON prompt**
- **不改策略链文件**（`strategies/{date}.json`）
- 盯盘发现的问题记问题清单，不改代码

### 3.5 变更安全规则

- **改完必验证**：`cronjob run` 或当天数据生成样图
- **双写**：改 CRON prompt 同时更新 `moni-sim-trading` skill
- **改 ≥3 个 CRON → 次版本号 +1**

## 四、发布管理

### 4.1 版本号规则

```
v<主版本>.<次版本>.<修订号>

主版本：架构重构、策略链大改     → 2.0
次版本：新功能、≥3 个 CRON 改动  → 1.1
修订号：单 CRON bugfix、脚本修复  → 1.0.1
```

### 4.2 同步机制

通过 `sync-to-git.sh` 一键同步 Hermes 运行态到 GitHub 仓库：

```bash
~/.hermes/scripts/sync-to-git.sh
```

做的事：
1. 从 Hermes 各目录 cp 最新文件到 git 仓库
2. 检查 diff，无变化则跳过
3. 自动导出当前 CRON prompt 到 `crons/`
4. 存档 Hermes 状态到 `.backup/YYYYMMDD_HHMM/`

### 4.3 Commit 规范

```
v<版本号>: <一句话总结>

- <改动1>
- <改动2>
```

### 4.4 发布 Checklist

```
[ ] 所有改动 CRON 已 cronjob run 验证
[ ] moni-sim-trading SKILL.md 已同步更新
[ ] changelog.md 已追加新版本条目
[ ] SRS 更新日志已记录完成
[ ] 迭代计划.md 已完成项打勾
[ ] sync-to-git.sh 执行无 diff 残留
[ ] git commit 含版本号 + 改动摘要
[ ] git tag v1.x.x
[ ] git push --tags
```

## 五、运维管理

### 5.1 每日盯盘三信号

| 信号 | 看什么 | 红灯条件 |
|------|------|:--:|
| CRON 健康 | `cronjob list` → `last_status` + `last_delivery_error` | 任一非 `ok` |
| 策略一致性 | `risk_state.level` vs `current_strategy.mode` | 不一致 |
| 硬约束穿透 | `execution_log` 中 `deviation` 非 null | 有偏离 |

三全绿 → 无需操作。红灯 → 按严重度处理：P0 立即修，P1/P2 记问题清单。

### 5.2 回滚流程

| 场景 | 操作 |
|------|------|
| 单 CRON prompt | 从 Git 历史取旧版 → `cronjob update` 回灌 |
| 脚本回滚 | `git checkout <tag> -- scripts/xxx.py` |
| 批量回滚 | 用 `.backup/` 快照恢复 |

**回滚安全原则**：盘中不回滚、单 CRON 优先、回滚也是版本（发新版本号）、Git tag 不可删。

### 5.3 回滚后必做

```
[ ] cronjob run 验证
[ ] changelog 标注 [REVERTED]
[ ] 根因分析 → 记入问题清单.md
[ ] 发修订版本号
```
