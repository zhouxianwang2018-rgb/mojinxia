# CRON 分层优化方案

> 2026-06-02 架构审视结论，待实施。

## 现状诊断

午盘和尾盘同时承担「侦察」和「执行」两个职责，导致侦察逻辑在 prompt 中复制了两份。

```
午盘检查 (10:00)                    尾盘检查 (14:35)
─────────────────                   ─────────────────
🔍 大盘/板块侦察                      🔍 大盘/板块侦察        ← 重复！
📸 保存快照                          📸 读快照对比           ← 互补
🚫 硬约束                            🚫 硬约束               ← 重复！
🎯 Scenario 匹配                     🎯 Scenario 匹配        ← 重复！
⚡ 执行交易（买入为主）                ⚡ 执行交易（防守为主）
✍️ 写 execution_log                 ✍️ 写 execution_log
                                     🚨 读午后预警           ← 尾盘独有
                                     📰 生成报纸             ← 尾盘独有
```

## 真正的分层

用户的直觉是对的，但午后检查的定位需要修正。午后检查并非「策略制定者」——它只写预警到 execution_log，不能改 scenarios、不能调 stop_loss。真正的分层：

```
策略层：收盘复盘（T-1 创建蓝图） + 盘前分析（T 日微调）     ← 唯一能改策略文件的
侦察层：午后检查（13:00 信息最丰富）                        ← 唯一做市场侦察的
执行层：午盘检查 + 尾盘检查                                 ← 只读侦察快照 → 硬约束 → 匹配执行
```

## 午后检查升级方案

从「只写预警文本」升级为「侦察 + 输出结构化市场快照 + 有限策略更新权」：

```json
{
  "market_snapshot": {
    "time": "13:16",
    "indices": {
      "sh_index": {"price": 3280, "change_pct": 0.15},
      "star50": {"price": 958, "change_pct": 1.66}
    },
    "sectors": {
      "semiconductor": {"change_pct": 1.03, "net_flow": -48, "trend": "V_reversal"},
      "consumer": {"change_pct": 0.50, "trend": "steady"}
    },
    "risk_signals": [
      {"type": "sector_swing", "detail": "半导体-1.85%→+1.03%, +2.88pp"},
      {"type": "position_alert", "detail": "无持仓，空仓状态"}
    ]
  },
  "strategy_updates": [
    {
      "action": "append_scenario",
      "scenario": {
        "if": "尾盘科创50守住+1.5%以上且半导体未回落",
        "then": "建仓半导体设备/材料，单票≤¥80,000",
        "type": "buy",
        "reason": "午后V反确认，板块资金回流"
      }
    }
  ]
}
```

## 执行层简化

午盘和尾盘 prompt 去掉独立侦察段落，改为：

```
Step 0: 加载策略链
Step 0a: 硬约束
Step 1: 加载{盘前/午后}市场快照（只读，不再独立侦察）
Step 2: 读 execution_log 中的前序操作
Step 3: Scenario 匹配 → 执行
Step 4: 写入 execution_log
```

尾盘额外步骤：读午后结构化快照中的 `strategy_updates`，将追加的 scenario 纳入匹配。

## 收益量化

| 指标 | 现状 | 优化后 |
|------|:---:|:---:|
| 独立 API 侦察调用/天 | 4 次 | 3 次 |
| 盘中策略更新能力 | 0 | 午后可追加 emergency/stop_loss/buy |
| 午→尾信息传递 | 原始文本→LLM 重新解读 | 结构化 JSON→直接匹配 |
| 午盘 prompt 行数 | ~300 | ~200 |
| 尾盘 prompt 行数 | ~400 | ~250 |

最大收益不是省 API 调用，而是午后从「只能喊不能动」升级为真正的盘中策略节点。

## 不改什么

- 午盘和尾盘两个交易窗口保持分开（10:00 vs 14:35 市场逻辑不同）
- 硬约束在两个执行点都保留（不能因侦察收敛放松风控）
- snapshot 机制保留（orders API 降级生命线）

## 交付格式优化（2026-06-02 第二次审视）

在侦察收敛（v5）基础上，进一步按"用户每天看到什么"精简交付：

### 优化前 → 优化后

| 时间 | CRON | 优化前 | 优化后 | 理由 |
|------|------|--------|--------|------|
| 09:00 | 盘前定调 | 📰报纸 | 📰报纸 | 不变——开盘前全景锚点 |
| 10:00 | 午盘执行 | 📝大报告（~20行，含大盘+板块+账户+熔断+结论） | 📝≤5行（风险+仓位+操作，无大盘/板块/账户） | 去重：大盘盘前已报、午后会更新；午盘价值只在"执行了什么" |
| 13:16 | 午后侦察 | 📝短文本 | 📝短文本 | 不变——格式已精炼 |
| 14:20 | 引擎一 | 📝推QQ（STAGE2/3候选表） | 🔇`local`（内部，`context_from`喂尾盘） | 用户不手动交易，看了也操作不了。尾盘后台自动消费 |
| 14:25 | 引擎二 | 🔇无交付 | 📰报纸（排名结果+交叉命中） | 排名结果是策略级信息，值得出图 |
| 14:35 | 尾盘执行 | 📰报纸 | 📝文本（操作+持仓+逻辑） | 尾盘本质是执行结果摘要，不需要整张报纸 |
| 15:05 | 收盘复盘 | 📰报纸 | 📰报纸 | 不变——全天最重要的信息密度锚点 |

### 用户每天收到

```
09:00  📰 盘前早参（报纸）
10:00  📝 午盘战报（≤5行）
13:16  📝 午后侦察（短文本）
14:25  📰 引擎二排名（报纸）
14:35  📝 尾盘战报（纯文本）
15:05  📰 收盘复盘（报纸）
```

从 6 条→还是 6 条，但每条价值明确不重复。午盘从 20 行砍到 5 行；引擎一静默、引擎二补位。
