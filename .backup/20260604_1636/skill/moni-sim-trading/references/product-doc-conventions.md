# 摸金虾 · 产品设计文档规范

> 2026-06-03 确立。后续新建 SRS 时严格遵循。

## 目录结构

```
~/.hermes/products/摸金虾/
├── README.md                   ← 项目入口，指向各文档
├── docs/                       ← 系统现状（"是什么"）
│   ├── architecture.md         ← ADR + 三体联动 + CRON 分层
│   └── development.md          ← 开发手册（需求→迭代→开发→发布→运维全生命周期）
├── roadmap/                    ← 规划层（"要做什么"）
│   ├── 需求清单.md              ← 轻量表格 + 约定
│   ├── 问题清单.md              ← 唯一问题追踪源
│   ├── 迭代计划.md              ← 唯一版本规划源（累积不清空）
│   └── 需求池/                  ← SRS 规格文件
└── changelog.md                ← 迭代计划衍生品（指向迭代计划，不单独维护）
```

## 需求管理

### 从想法到规格

```
想法 ──→ 需求清单.md（分配#号，状态=待排期）
              │
              ├── 开始讨论设计 ──→ 需求池/SRS-NNN（状态=设计中）
              │                         │
              │                    方案定稿（状态=待实施）
              │                         │
              │              迭代计划选中（状态=实施中）
              │                         │
              │                  实施完成 → SRS 更新日志闭合（状态=已完成）
              │
              └── 讨论后关闭 ──→ 状态=已关闭
```

关键点：需求清单#号 = SRS 编号（同一体系）；SRS 在方案讨论时就可以建，不等迭代选中。

### 需求清单条目格式

```
| # | 状态 | 优先级 | 需求 | 关闭条件 | 关联 | 创建 |
```

状态枚举：待排期 / 设计中 / 待实施 / 实施中 / 已完成 / 已关闭。状态与 SRS 生命周期同步。

## 问题管理

### 唯一问题源

```
盯盘红灯 / 用户提出 / 审计报警 ──→ 问题清单.md
                                         │
                                   按严重度分流：
                                   P0 → 立即修，关条目
                                   P1 → 周六修，关条目
                                   P2 简单 → 周六修，关条目
                                   P2 需设计 → 关条目，同时建需求清单条目 → SRS
```

所有问题先登记问题清单。没有第二个地方。

### 严重度定义

| 等级 | 含义 | 响应 |
|:--:|------|:--:|
| 🔴 P0 | 影响当天交易正确性 | **立即修** |
| 🟡 P1 | 影响近期交易质量 | 周六集中修 |
| 🟢 P2 | 体验/远期改进 | 周六修；需设计则走需求流程 |

### 问题清单条目格式

```
| # | 状态 | 严重 | 问题 | 现象 | 来源 | 关联 |
```

状态枚举：待排期 / 已排期 / 已关闭。来源枚举：盯盘 / 审计 / 用户。

## SRS 文件规范

### 命名

`需求池/SRS-NNN-description.md`，NNN = 需求清单#号，三位补零。

### 五段固定模板

```
# 标题
> SRS-NNN · 创建日期 · 状态

## 一、业务需求 — 问题 + 目标 + 成功指标
## 二、方案设计 — 架构/数据/关键决策
## 三、风险与缓解
## 四、关联 — 与其它 Feature 的依赖/冲突
## 五、更新日志 — 日期 + 变更摘要
```

### SRS 生命周期

| 状态 | 含义 | 何时进入 |
|------|------|------|
| 设计中 | 方案讨论，未定稿 | 需求条目开始讨论时 |
| 待实施 | 方案定稿，等排期 | 方案设计完成、风险已评估 |
| 实施中 | 本轮迭代在做 | 迭代计划选中 |
| 已完成 | 上线 | 发布 checklist 通过 |

### 设计原则

- 生命周期字段与扫描快照分离
- 不存可推导字段（如 total_active 由计数得出）
- 不做预留空壳（用到再扩）
- 方案设计不写分阶段实施（实施计划在迭代计划中管理）

## 迭代计划格式

迭代计划是**唯一的版本规划源**——声明当前在做哪个版本、放什么、什么时候上线。
累积不清空，已完成的版本保留在下方（标记 ✅），需要发产品更新时从中提取。

```
# 迭代计划

## v1.1 · 主题 🔄
> 目标：一句话。

### 时间线
| 节点 | 预计 |
|:--|:--:|
| 方案定稿 | M/D |
| 实施完成 | M/D |
| 上线 | M/D |

### 工作项
| 来源 | 编号 | 事项 | 现状 |
|:--:|:--:|------|------|
| 📋 | 001 | 需求名 | Phase 0 验证 |
| 🐛 | 002 | 问题名 | 待修复 |

---

## v1.0 · 首个正式版本 ✅
> M/D 上线

上线总结一段话。

| 来源 | 编号 | 事项 |
|:--:|:--:|------|
| 📋 | — | xxx |
| 🐛 | — | xxx |
```

**约定：**
- 版本号驱动（v1.0, v1.1...），不用"迭代 N"
- 来源用 emoji 前缀：📋需求 / 🐛问题，不按来源分组
- 进行中版本有「现状」列，已完成版本去掉该列
- 时间线三个节点：方案定稿 / 实施完成 / 上线
- changelog.md 指向迭代计划，不单独维护
