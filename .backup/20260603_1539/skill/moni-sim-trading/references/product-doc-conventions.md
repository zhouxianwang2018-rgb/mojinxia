# 摸金虾 · 产品设计文档规范

> 2026-06-03 确立。后续新建 Feature/需求时严格遵循。

## 目录结构

```
~/.hermes/products/摸金虾/
├── README.md                   ← 项目入口，指向各文档
├── docs/                       ← 系统现状（"是什么"）
│   ├── architecture.md         ← ADR + 三体联动 + CRON 分层
│   └── development.md          ← 开发手册
├── roadmap/                    ← 规划层（"要做什么"）
│   ├── 需求清单.md              ← 轻量表格 + 约定（命名/结构/流程）
│   ├── 问题清单.md              ← Bug/瓶颈表
│   ├── 迭代计划.md              ← 从两清单挑选本轮要做的
│   └── 需求池/                  ← 每条需求的完整 SRS
│       ├── SRS-001-...
│       └── SRS-002-...
└── changelog.md                ← 全局版本日志（里程碑级）
```

## 三层文件关系

| 文件 | 存什么 | 定位 |
|------|------|------|
| `需求清单.md` | 表格：状态/优先级/描述/关闭条件/关联 SRS | 轻量 stub，不写方案设计 |
| `问题清单.md` | 表格：状态/严重/问题/现象/关联 | Bug 和瓶颈追踪 |
| `需求池/SRS-NNN-*.md` | 五段完整规格 | 完整需求规格说明书，方案设计在此展开 |

**流转**：需求清单选中 → 建 SRS 文件 → 需求清单状态更新为"已排期 · SRS-NNN"。

## SRS 文件规范

### 命名

`SRS-NNN-description.md`，NNN 为三位递增序号。

### 五段固定模板

```
# 标题
> SRS-NNN · 创建日期 · 状态

## 一、业务需求
### 问题
### 目标
### 成功指标

## 二、方案设计
### 2.x ...

## 三、风险与缓解
| 风险 | 缓解 |

## 四、关联
| Feature | 关系 |

## 五、更新日志
| 日期 | 变更 |
```

**设计原则**：
- 生命周期字段与扫描快照分离
- 不存可推导字段
- 不做预留空壳（用到再扩）
- 方案设计不做分阶段实施（实施计划在迭代计划中管理）

## 需求清单条目格式

```
| # | 状态 | 优先级 | 需求 | 关闭条件 | 关联 | 创建 |
```

状态枚举：待排期 / 已排期 / 已升级 / 已关闭。

## 问题清单条目格式

```
| # | 状态 | 严重 | 问题 | 现象 | 关联 |
```

严重枚举：🔴高 / 🟡中 / 🟢低。

## 迭代计划格式

```
# 迭代 N · 日期范围

## 来自需求清单
- [ ] xxx → SRS-NNN

## 来自问题清单
- [ ] xxx → 直接修 / SRS-NNN

## 本周不排
- xxx（原因）
```
